# 需求分析文档 (Requirements Analysis)

## 1. 赛题背景

**赛题**: Level 3 - 产品经理 Agent：用户痛点和应用场景挖掘，输出需求文档。
**目标**: 构建一个 Agent，能够处理多源、非结构化的用户反馈，通过逻辑推理和商业洞察，输出结构化的痛点分析和标准 PRD 文档。
**核心模型**: DeepSeek-V3.2

## 2. 核心任务拆解

### 2.1 输入处理 (Input Processing)

Agent 需要接收并理解以下两类信息：

1. **产品基础信息**:
    - 产品名称 (Name)
    - 产品定位/描述 (Description)
    - 核心功能列表 (Core Features) - _隐含需求，需从描述中提取或作为背景知识_
2. **多源用户反馈 (Feedback Data)**:
    - 来源 (Source): AppStore, 客服, 访谈等。
    - 用户层级 (User Level): VIP, 新用户, 活跃用户等。
    - 内容 (Content): 自然语言描述，可能包含情绪、拼写错误、模糊表述。
    - 时间戳 (Timestamp): 用于判断反馈的时效性。

### 2.2 核心推理链路 (Reasoning Chain)

Agent 必须具备以下思维能力：

1. **去噪与清洗**: 识别并过滤纯情绪宣泄、无意义内容。
2. **痛点挖掘 (Root Cause Analysis)**:
    - _Surface_: 用户说什么（例如：“加载太慢”）。
    - _Deep_: 用户真正想要什么（例如：“社交炫耀受挫”，“即时满足感缺失”）。
3. **场景建模 (Scenario Modeling)**:
    - Who: 谁在用？
    - When: 什么时间/时机？
    - Where: 在什么界面/环境下？
    - What: 遇到了什么具体阻碍？
4. **优先级排序 (Prioritization)**:
    - 使用 ICE (Impact, Confidence, Ease) 或 RISE 模型。
    - 综合考虑：用户量级、频率、商业价值。

### 2.3 输出生成 (Output Generation)

输出需包含两个部分：

1. **痛点分析报告 (JSON)**:
    - 核心痛点 (Pain Point)
    - 根本原因 (Root Cause)
    - 场景描述 (Scenario)
    - 优先级 (Priority)
2. **标准 PRD 文档 (Markdown/JSON)**:
    - 文档标题
    - 项目背景 (Background)
    - 用户故事 (User Stories)
    - 功能需求 (Functional Requirements) - 需包含 ID, 名称, 描述, 验收标准。
    - 数据指标 (Data Metrics)

## 3. 评分标准对应策略

| 评分维度           | 权重 | 应对策略 (DeepSeek-V3.2)                                                                               |
| :----------------- | :--- | :----------------------------------------------------------------------------------------------------- |
| **痛点归因准确度** | 30%  | 利用 DeepSeek 强大的逻辑推理能力，设计 CoT (思维链) Prompt，强制模型进行“现象->原因->本质”的三层推导。 |
| **场景建模能力**   | 20%  | 在 Prompt 中要求模型进行“角色扮演”，还原用户操作路径，补全缺失的上下文信息。                           |
| **解决方案可行性** | 20%  | 注入产品设计原则知识库（作为 System Prompt），确保方案不脱离技术和商业实际。                           |
| **PRD 结构规范**   | 30%  | 严格定义输出 JSON Schema，利用模型的指令遵循能力确保格式精确。                                         |

## 4. 难点与风险

1. **幻觉问题**: 模型可能会臆造不存在的功能或反馈。 -> _对策: 限制模型仅基于提供的 Context 回答，并在 Prompt 中强调“基于事实”。_
2. **优先级判断主观性**: 不同的 PM 可能有不同判断。 -> _对策: 要求模型输出评分依据 (Rationale)，使决策透明化。_
3. **Token 限制**: 如果反馈数据量极大。 -> _对策: 采用分批处理 (Map-Reduce) 或 关键信息提取 (Summary) 策略。_
